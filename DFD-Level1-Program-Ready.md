# DFD Level 1 ตามโปรแกรมจริงของระบบ

ไฟล์นี้เขียนจาก **โปรแกรมจริงที่อยู่ในโปรเจกต์** ไม่ได้อิงจาก DFD เดิมอย่างเดียว  
ใช้เป็นต้นแบบสำหรับแก้เอกสารและแก้ผัง DFD Level 1 ให้ตรงกับระบบที่พัฒนาจริง

---

## 1. ก่อนอื่น ต้องเข้าใจให้ตรงกัน

จากโปรแกรมจริงใน `backend/src/routes` ระบบของเรามีงานหลักดังนี้

- สมัครสมาชิก / เข้าสู่ระบบ
- ดูข้อมูลคลินิกและบริการ
- จัดการข้อมูลเจ้าของสัตว์เลี้ยง
- จัดการข้อมูลสัตว์เลี้ยง
- จัดการนัดหมาย
- บันทึกการรักษา
- บันทึกผ่าตัด
- บันทึกวัคซีน
- ออกใบเสร็จ / รับชำระเงิน
- บันทึกรายจ่าย
- ออกรายงาน

และจากโปรแกรมจริงตอนนี้:

- `user` **ไม่ได้จองนัดเอง**
- `admin / เจ้าหน้าที่คลินิก` เป็นคนสร้าง แก้ไข ยืนยัน ยกเลิกนัดหมาย
- `user` เข้ามาได้แค่ดูนัดหมายของตัวเอง
- การรักษาเชื่อมกับ `tb_treatment` และ `tb_treatment_detail`
- ใบเสร็จเชื่อมกับ `tb_receipt` และ `tb_receipt_detail`
- การรักษา / ผ่าตัด / วัคซีน ใช้ `vet_id` อ้างอิงสัตวแพทย์

---

## 2. ทำไม DFD เดิมถึงโดนบอกว่ายังคล้าย DFD 0

ปัญหาไม่ได้อยู่ที่ “อยู่หน้าเดียว” อย่างเดียว แต่หลักๆ อยู่ที่ **รูปแบบ**

### 2.1 process ย่อยเยอะ แต่ความหมายการไหลของข้อมูลยังไม่ balance

ในไฟล์เดิมมีทั้ง

- `1.1 1.2 1.3`
- `2.1 2.2`
- `3.1 3.2`
- ...
- `14.1 14.2`

รวมกันในหน้าเดียว และแต่ละกลุ่มมี data flow จำนวนมาก  
ทำให้อ่านแล้วไม่ชัดว่า “นี่คือ Level 1 ของ process ไหนกันแน่”

### 2.2 บางส่วนยังดูเป็นภาพรวมมากเกินไป

ตัวอย่างเช่น flow ชื่อกว้างๆ แบบ

- ข้อมูลการรักษา
- ผลการจัดการข้อมูล
- ข้อมูลการนัดหมาย

ถ้าอาจารย์ต้องการ DFD Level 1 จริง มักจะอยากเห็นว่า

- รับอะไรเข้า
- ตรวจสอบอะไร
- บันทึกลง data store ไหน
- ส่งผลลัพธ์อะไรออก

แบบละเอียดและสมดุลกว่านี้

### 2.3 ตอนนี้ผังเดิมเป็นกึ่งผสมระหว่าง “ภาพรวมทั้งระบบ” กับ “child process”

สรุปสั้นๆ:

- ถ้าจะเป็น **ภาพรวมทั้งระบบ** ควรใช้ระดับ `1.0 - 14.0`
- ถ้าจะเป็น **การแตกย่อย** ควรเลือกแตกทีละ process เช่น `6.0` เป็น `6.1 6.2 6.3`

เพราะงั้น ถ้าจะเอาให้ชัดและส่งอาจารย์ง่ายที่สุด  
**DFD Level 1 ของระบบ ควรเขียนเป็นระดับ 1.0 - 14.0 ก่อน**

## 3. External Entity ที่ควรใช้

ใช้ 3 หน่วยงานภายนอกหลัก

1. `ผู้ใช้งานทั่วไป`
2. `เจ้าของสัตว์เลี้ยง`
3. `ผู้ดูแลระบบ / เจ้าหน้าที่คลินิก`

### ความหมาย

- `ผู้ใช้งานทั่วไป` = คนที่ยังไม่ login แต่ดูข้อมูลคลินิก/บริการ และสมัครสมาชิกได้
- `เจ้าของสัตว์เลี้ยง` = ผู้ใช้ที่ login แล้ว ใช้ดูข้อมูลของตัวเอง
- `ผู้ดูแลระบบ / เจ้าหน้าที่คลินิก` = คนที่จัดการข้อมูลหลักของระบบ

## 5. DFD Level 1 ที่ควรเป็นตามโปรแกรมเรา

ตรงนี้คือ **เวอร์ชันที่แนะนำให้ใช้แก้ผัง**

### 1.0 แสดงข้อมูลคลินิกและบริการ

หน้าที่:

- รับคำขอดูข้อมูลคลินิก
- ดึงข้อมูลคลินิก
- ดึงข้อมูลบริการ
- ส่งผลให้ผู้ใช้งานทั่วไปและเจ้าของสัตว์เลี้ยง

เชื่อมกับ:

- `ผู้ใช้งานทั่วไป`
- `เจ้าของสัตว์เลี้ยง`
- `D1 ข้อมูลคลินิก`
- `D13 ข้อมูลบริการและราคา`

### 2.0 สมัครสมาชิก

หน้าที่:

- รับข้อมูลสมัครสมาชิก
- ตรวจสอบข้อมูลซ้ำ
- บันทึกผู้ใช้ใหม่

เชื่อมกับ:

- `ผู้ใช้งานทั่วไป`
- `D2 ข้อมูลผู้ใช้ระบบ`
- `D3 ข้อมูลเจ้าของสัตว์เลี้ยง`

### 3.0 เข้าสู่ระบบ

หน้าที่:

- รับชื่อผู้ใช้และรหัสผ่าน
- ตรวจสอบบัญชี
- ตรวจสอบสิทธิ์การใช้งาน
- ส่งผลการเข้าสู่ระบบ

เชื่อมกับ:

- `ผู้ใช้งานทั่วไป`
- `เจ้าของสัตว์เลี้ยง`
- `ผู้ดูแลระบบ / เจ้าหน้าที่คลินิก`
- `D2 ข้อมูลผู้ใช้ระบบ`

### 4.0 จัดการข้อมูลเจ้าของสัตว์เลี้ยง

หน้าที่:

- เพิ่ม / แก้ไข / ค้นหา ข้อมูลเจ้าของสัตว์เลี้ยง
- เชื่อมกับบัญชีผู้ใช้

เชื่อมกับ:

- `เจ้าของสัตว์เลี้ยง`
- `ผู้ดูแลระบบ / เจ้าหน้าที่คลินิก`
- `D2 ข้อมูลผู้ใช้ระบบ`
- `D3 ข้อมูลเจ้าของสัตว์เลี้ยง`

หมายเหตุ:

- ในโปรแกรมจริงไม่ใช้ `owner_address` แล้ว

### 5.0 จัดการข้อมูลสัตว์เลี้ยง

หน้าที่:

- เพิ่ม / แก้ไข / ค้นหา / ลบ ข้อมูลสัตว์เลี้ยง
- ตรวจสอบว่าเป็นสัตว์ของเจ้าของคนใด

เชื่อมกับ:

- `เจ้าของสัตว์เลี้ยง`
- `ผู้ดูแลระบบ / เจ้าหน้าที่คลินิก`
- `D3 ข้อมูลเจ้าของสัตว์เลี้ยง`
- `D4 ข้อมูลสัตว์เลี้ยง`

### 6.0 จัดการข้อมูลการนัดหมาย

หน้าที่:

- เจ้าหน้าที่สร้างนัดหมาย
- แก้ไขวันและเวลา
- ยืนยัน / ยกเลิกนัดหมาย
- เจ้าของสัตว์เลี้ยงดูนัดหมายของตัวเอง
- ส่งอีเมลแจ้งเตือน

เชื่อมกับ:

- `เจ้าของสัตว์เลี้ยง`
- `ผู้ดูแลระบบ / เจ้าหน้าที่คลินิก`
- `D3 ข้อมูลเจ้าของสัตว์เลี้ยง`
- `D4 ข้อมูลสัตว์เลี้ยง`
- `D5 ข้อมูลการนัดหมาย`
- `D1 ข้อมูลคลินิก` (ใช้ประกอบอีเมลแจ้งเตือน)

หมายเหตุสำคัญ:

- process นี้ในระบบจริง **ไม่ใช่การจองของ user**
- ต้องเขียนให้สอดคล้องกับโปรแกรมว่า **คลินิกเป็นผู้จัดการนัดหมาย**

### 7.0 บันทึกการรักษาและหัตถการ

หน้าที่:

- เลือกสัตว์เลี้ยง
- บันทึกอาการและการวินิจฉัย
- ระบุสัตวแพทย์
- เลือกรายการบริการ

- บันทึกข้อมูลการรักษา
- บันทึกรายละเอียดการรักษา

เชื่อมกับ:

- `ผู้ดูแลระบบ / เจ้าหน้าที่คลินิก`
- `D4 ข้อมูลสัตว์เลี้ยง`
- `D6 ข้อมูลการรักษา`
- `D13 ข้อมูลบริการและราคา`
- `D14 ข้อมูลสัตวแพทย์`

### 8.0 บันทึกประวัติวัคซีน

หน้าที่:

- บันทึกประวัติวัคซีน

### 9.0 บันทึกการผ่าตัด

หน้าที่:

- ระบุรายละเอียดการผ่าตัด
- ระบุสัตวแพทย์
- บันทึกข้อมูลการผ่าตัด

เชื่อมกับ:

- `ผู้ดูแลระบบ / เจ้าหน้าที่คลินิก`

### 10.0 บันทึกการชำระเงินและออกใบเสร็จ

หน้าที่:

- สร้างใบเสร็จ
- บันทึกรายการย่อยของใบเสร็จ
- อัปเดตสถานะการชำระเงิน
- แนบหลักฐานการโอนเงินได้

เชื่อมกับ:

- `เจ้าของสัตว์เลี้ยง`
- `ผู้ดูแลระบบ / เจ้าหน้าที่คลินิก`
- `D3 ข้อมูลเจ้าของสัตว์เลี้ยง`
- `D6 ข้อมูลการรักษา`
- `D9 ข้อมูลใบเสร็จรับเงิน`

หมายเหตุ:

- ถ้าในเอกสารยังใช้ใบเสร็จ data store เดียว ต้องแก้

### 12.0 ออกรายงานสรุปการรักษา

- ดึงข้อมูลการรักษา วัคซีน ผ่าตัด
- ประมวลผลและจัดรูปแบบรายงาน

เชื่อมกับ:

- `ผู้ดูแลระบบ / เจ้าหน้าที่คลินิก`
- `D6 ข้อมูลการรักษา`
- `D7 ข้อมูลการผ่าตัด`
- `D8 ข้อมูลวัคซีน`

### 13.0 ออกรายงานสรุปรายรับ-รายจ่าย

หน้าที่:

- รับช่วงเวลา

## 6. ถ้าอาจารย์ถามว่า “แล้ว DFD Level 1 ของคุณต่างจากของเดิมยังไง”

ให้ตอบแบบนี้ได้เลย

> เดิมผังที่จัดทำมีลักษณะผสมระหว่างภาพรวมของระบบกับการแตกย่อยกระบวนการหลายส่วนพร้อมกัน ทำให้ยังมองไม่ชัดว่าแต่ละ process หลักของระบบเชื่อมกับ data store และ external entity อย่างไร ในการปรับปรุงครั้งนี้จึงได้จัดโครงสร้าง DFD Level 1 ใหม่ให้สอดคล้องกับโปรแกรมจริง โดยกำหนด process หลักในระดับ 1.0 - 14.0 ให้ครอบคลุมการสมัครสมาชิก การเข้าสู่ระบบ การจัดการข้อมูลเจ้าของสัตว์เลี้ยง การจัดการข้อมูลสัตว์เลี้ยง การนัดหมาย การรักษา วัคซีน การผ่าตัด การเงิน รายจ่าย และรายงาน พร้อมทั้งปรับ data store ให้ตรงกับฐานข้อมูลจริงของระบบ

## 7. ถ้าจะทำ child process ต่อ ให้เลือก process สำคัญเหล่านี้

ถ้าอาจารย์ต้องการ “แตกย่อยต่อ” ให้เลือก process ที่สำคัญที่สุดในโปรแกรม

### 6.0 จัดการข้อมูลการนัดหมาย

## 8. ข้อสรุปที่ควรยึดเวลาแก้เอกสาร

- DFD ต้องอิง **โปรแกรมจริง**
- ฝั่ง user **ไม่ได้จองนัดเอง**
- นัดหมายเป็นงานของ `admin / เจ้าหน้าที่คลินิก`
- ระบบมี data store จริง 15 ตัว
- การรักษาต้องมี `tb_treatment_detail`
- การเงินต้องมี `tb_receipt_detail`
- การรักษา / ผ่าตัด / วัคซีน ต้องเชื่อม `ข้อมูลสัตวแพทย์`

## 9. ถ้าจะวาดส่งอาจารย์แบบเร็วที่สุด

แนะนำให้วาดแบบนี้

1. วาง external entity 3 ตัวทางซ้าย
2. วาง process `1.0 - 14.0` ตรงกลาง
3. วาง data store `D1 - D15` ทางขวา
4. ลากเส้นเฉพาะที่เกี่ยวกันจริง
5. อย่าใส่ process ย่อย `1.1, 1.2, 1.3 ...` ปนในภาพนี้

ภาพนี้จะทำหน้าที่เป็น **DFD Level 1 ของระบบตามโปรแกรมจริง**
