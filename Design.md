# Pet Clinic Management System Design

## 1. ภาพรวมโครงการ

โปรแกรมนี้เป็นระบบจัดการคลินิกสัตว์เลี้ยงแบบเว็บแอปพลิเคชัน สำหรับใช้บริหารงานหน้าคลินิกและให้เจ้าของสัตว์เลี้ยงเข้ามาติดตามข้อมูลของตนเองได้ผ่านระบบออนไลน์ โดยแบ่งการใช้งานออกเป็น 2 ฝั่งหลักคือ

- `Admin` สำหรับพนักงานหรือผู้ดูแลระบบคลินิก
- `User` สำหรับเจ้าของสัตว์เลี้ยง

ระบบถูกพัฒนาด้วยแนวคิดให้ข้อมูลทุกส่วนเชื่อมกันเป็น flow เดียว เช่น สัตว์เลี้ยงเชื่อมกับเจ้าของ, การรักษาเชื่อมกับสัตวแพทย์และบริการ, ใบเสร็จเชื่อมกับการรักษา, และ dashboard/report ดึงข้อมูลจากงานจริงของคลินิกมาสรุปผล

---

## 2. วัตถุประสงค์ของระบบ

### 2.1 วัตถุประสงค์หลัก

- จัดเก็บข้อมูลเจ้าของสัตว์และสัตว์เลี้ยงอย่างเป็นระบบ
- รองรับการนัดหมายเข้ารับบริการ
- รองรับการบันทึกการรักษา ผ่าตัด และวัคซีน
- รองรับการออกใบเสร็จและติดตามการชำระเงิน
- รองรับการบันทึกรายจ่ายของคลินิก
- สรุปผลการดำเนินงานผ่าน dashboard และรายงาน

### 2.2 วัตถุประสงค์เชิงการใช้งาน

- ลดการใช้เอกสารกระดาษ
- ลดการค้นหาข้อมูลย้อนหลังแบบ manual
- ทำให้เจ้าของสัตว์เข้าถึงข้อมูลของตนเองได้ง่ายขึ้น
- ช่วยผู้ดูแลคลินิกเห็นภาพรวมรายรับ รายจ่าย และงานประจำวัน

## 3. กลุ่มผู้ใช้งาน

### 3.1 Admin

สิทธิ์หลัก

- จัดการเจ้าของสัตว์
- จัดการสัตว์เลี้ยง

- จัดการข้อมูลคลินิก
- จัดการนัดหมาย
- จัดการบริการและราคา
- จัดการการรักษา
- จัดการการผ่าตัด
- จัดการการฉีดวัคซีน
- จัดการใบเสร็จ
- จัดการรายจ่าย
- ดู dashboard และรายงาน

### 3.2 User

- ดูใบเสร็จและสถานะการชำระเงิน

## 4. ขอบเขตของระบบ

ระบบในเวอร์ชันนี้ครอบคลุมกระบวนการหลักของคลินิกสัตว์เลี้ยงดังนี้

1. การสมัครสมาชิกและเข้าสู่ระบบ
2. การจัดการข้อมูลเจ้าของสัตว์
3. การจัดการข้อมูลสัตว์เลี้ยง
4. การนัดหมาย
5. การจัดการบริการและราคาค่ารักษา
6. การบันทึกการรักษา
7. การบันทึกการผ่าตัด
8. การบันทึกวัคซีน
9. การออกใบเสร็จและรับชำระเงิน
10. การจัดการรายจ่ายคลินิก
11. การแสดงผล dashboard และรายงาน

## 5. สถาปัตยกรรมระบบ

ระบบใช้โครงสร้างแบบ 3 ส่วนหลัก

### 5.1 Frontend

- พัฒนาโดย `Vue 3`
- ใช้ `Vue Router`
- ใช้ `Axios` สำหรับเรียก API
- แยกหน้าฝั่ง `user` และ `admin`

### 5.2 Backend

- พัฒนาโดย `Node.js + Express`
- ใช้ JWT สำหรับตรวจสอบสิทธิ์
- แยก route ตามหน้าที่ของโมดูล
- มี logic สำหรับตรวจสอบสิทธิ์ user/admin
- รองรับการส่งอีเมลแจ้งเตือนนัดหมาย

### 5.3 Database

- ใช้ `PostgreSQL`
- เก็บข้อมูลเชิงสัมพันธ์ระหว่าง user, owner, pet, appointment, treatment, receipt และข้อมูลคลินิกอื่น ๆ

## 7. โมดูลหลักของระบบ

### 7.1 Authentication Module

หน้าที่

- สมัครสมาชิก
- เข้าสู่ระบบ
- แยกสิทธิ์ `admin` และ `user`
- ป้องกันการเข้าหน้าไม่ตรง role

ข้อมูลสำคัญ

- `tb_user`

### 7.2 Owner Module

หน้าที่

- เก็บข้อมูลเจ้าของสัตว์
- เชื่อมกับบัญชีผู้ใช้
- ใช้เป็นจุดอ้างอิงกลางสำหรับสัตว์เลี้ยง ใบเสร็จ และการติดต่อ

ข้อมูลสำคัญ

- `tb_owner`

### 7.3 Pet Module

หน้าที่

- เก็บข้อมูลสัตว์เลี้ยง
- เชื่อมกับเจ้าของสัตว์
- ใช้ในนัดหมาย การรักษา วัคซีน และผ่าตัด

ข้อมูลสำคัญ

- `tb_pet`

### 7.4 Appointment Module

หน้าที่

- นัดวันและเวลาสำหรับสัตว์เลี้ยง
- ตรวจสอบเวลาซ้ำ
- ยืนยันหรือยกเลิกนัด
- ส่งอีเมลแจ้งเตือนเจ้าของสัตว์เลี้ยง

ข้อมูลสำคัญ

- `tb_appointment`

### 7.5 Service Module

หน้าที่

- จัดการรายการบริการ
- กำหนดราคามาตรฐาน
- ใช้เป็นรายการอ้างอิงในการรักษา

ข้อมูลสำคัญ

- `tb_service`

### 7.6 Treatment Module

หน้าที่

- บันทึกอาการ
- บันทึกการวินิจฉัย
- บันทึกรายการบริการที่ใช้รักษา
- คำนวณยอดรวมค่ารักษา
- เชื่อมกับสัตวแพทย์

ข้อมูลสำคัญ

- `tb_treatment`
- `tb_treatment_detail`

### 7.7 Surgery Module

หน้าที่

- บันทึกการผ่าตัด
- เก็บชนิดการผ่าตัด ผลการรักษา ยาสลบ และสัตวแพทย์ผู้รับผิดชอบ

ข้อมูลสำคัญ

- `tb_surgery`

### 7.8 Vaccine Module

หน้าที่

- บันทึกประวัติการฉีดวัคซีน
- เก็บชื่อวัคซีน เลข lot วันฉีด และสัตวแพทย์ผู้ฉีด

ข้อมูลสำคัญ

- `tb_vaccine_rec`

### 7.9 Receipt & Payment Module

หน้าที่

- สร้างใบเสร็จจากการรักษา
- แสดงสถานะชำระเงิน
- รองรับเงินสดและโอนเงิน
- รองรับแนบหลักฐานการโอน

ข้อมูลสำคัญ

- `tb_receipt`
- `tb_receipt_detail`

### 7.10 Expense Module

หน้าที่

- บันทึกรายจ่ายของคลินิก
- จัดกลุ่มหมวดหมู่รายรับ/รายจ่าย
- ใช้ใน dashboard และ reports

ข้อมูลสำคัญ

- `tb_category`
- `tb_expense`

### 7.11 Clinic Module

หน้าที่

- เก็บชื่อคลินิก
- เก็บเบอร์โทร
- เก็บที่อยู่
- เก็บเวลาเปิดทำการ
- ใช้แสดงในหน้าเว็บและเอกสาร

ข้อมูลสำคัญ

- `tb_clinic`

### 7.12 Veterinarian Module

หน้าที่

- เก็บข้อมูลสัตวแพทย์
- ใช้เชื่อมกับการรักษา ผ่าตัด และวัคซีน

ข้อมูลสำคัญ

- `tb_veterinarian`

### 7.13 Dashboard & Reports Module

หน้าที่

- สรุปจำนวนนัดหมาย
- สรุปรายรับ
- สรุปรายจ่าย
- สรุปกำไรสุทธิ
- แสดงรายงานตามเดือน/ปี

ข้อมูลสำคัญ

- ใช้ข้อมูลจาก `tb_appointment`, `tb_receipt`, `tb_expense`, `tb_treatment`

## 9. ความสัมพันธ์ของข้อมูล

โครงสร้างความสัมพันธ์หลักของระบบ

- `tb_user` 1:1 หรือ 1:many กับ `tb_owner`
- `tb_owner` 1:many กับ `tb_pet`
- `tb_pet` 1:many กับ `tb_appointment`
- `tb_pet` 1:many กับ `tb_treatment`
- `tb_treatment` 1:many กับ `tb_treatment_detail`
- `tb_service` 1:many กับ `tb_treatment_detail`
- `tb_treatment` 1:1 หรือ 1:many เชิงเอกสารกับ `tb_receipt`
- `tb_receipt` 1:many กับ `tb_receipt_detail`
- `tb_category` 1:many กับ `tb_expense`
- `tb_veterinarian` เชื่อมกับ `tb_treatment`, `tb_surgery`, `tb_vaccine_rec`

## 10. Flow การทำงานของระบบ

### 10.1 Flow ฝั่ง User

1. สมัครสมาชิกหรือเข้าสู่ระบบ
2. เข้าหน้าข้อมูลส่วนตัว
3. ดูข้อมูลสัตว์เลี้ยงของตนเอง
4. ดูนัดหมาย
5. ดูใบเสร็จ
6. ดูประวัติการรักษา

### 10.2 Flow ฝั่ง Admin

1. เข้าสู่ระบบ
2. จัดการข้อมูลพื้นฐาน
   - สมาชิก
   - เจ้าของสัตว์
   - สัตว์เลี้ยง
   - สัตวแพทย์
   - ข้อมูลคลินิก
3. สร้างนัดหมาย
4. บันทึกการรักษา
5. สร้างใบเสร็จจากการรักษา
6. รับชำระเงิน
7. บันทึกรายจ่าย
8. ตรวจ dashboard และรายงาน

## 11. กฎธุรกิจสำคัญ

- ผู้ใช้ทั่วไปเข้าถึงได้เฉพาะข้อมูลของตนเอง
- ผู้ดูแลระบบเข้าถึงข้อมูลรวมของคลินิกได้
- การนัดหมายต้องมีสัตว์เลี้ยงอ้างอิงเสมอ
- การรักษาต้องผูกกับสัตว์เลี้ยง
- การรักษาสามารถมีหลายรายการบริการ
- ใบเสร็จเชื่อมกับการรักษา
- รายงานทางการเงินอ้างอิงจากใบเสร็จและรายจ่าย
- สัตวแพทย์ควรเชื่อมกับการรักษา ผ่าตัด และวัคซีน

## 12. แนวคิดการออกแบบเว็บไซต์

ส่วนนี้เป็นแนวทางการตกแต่งเว็บไซต์ของโปรแกรมเพื่อให้ภาพรวมของระบบดูเป็นงานจริง มีความเป็นคลินิกสมัยใหม่ และใช้งานง่าย

### 12.1 แนวคิดภาพรวม

เว็บไซต์ควรให้ความรู้สึกดังนี้

- สะอาด
- น่าเชื่อถือ
- เป็นมิตร
- ดูเป็นระบบงานจริง
- ใช้งานง่ายทั้งเจ้าหน้าที่และเจ้าของสัตว์

ไม่ควรออกแบบให้เหมือนหน้า landing page ขายของ แต่ควรเป็น `application UI` ที่เน้นการทำงานจริง

## 13. แนวทาง Visual Design

### 13.1 Mood & Tone

- `Professional` แต่ไม่แข็ง
- `Soft medical` คล้ายคลินิกหรือ health service
- `Friendly` เพื่อให้เข้ากับงานสัตว์เลี้ยง

### 13.2 โทนสีหลัก

แนะนำ palette หลัก

- Primary: `#4F46E5` หรือ `#4338CA`
- Secondary: `#0EA5E9`
- Success: `#10B981`
- Warning: `#F59E0B`
- Danger: `#EF4444`
- Background: `#F8FAFC`
- Surface: `#FFFFFF`
- Text Main: `#0F172A`
- Text Sub: `#64748B`

เหตุผล

- น้ำเงิน/ม่วงให้ความรู้สึกน่าเชื่อถือและทันสมัย
- เขียวใช้กับสถานะสำเร็จและข้อมูลสุขภาพ
- แดงใช้กับการยกเลิกหรือความผิดพลาด

### 13.3 Typography

แนะนำ

- ใช้ฟอนต์ไทยที่อ่านง่าย เช่น `Sarabun` หรือ `Noto Sans Thai`
- หัวข้อใช้ weight 700-800
- ข้อความทั่วไปใช้ weight 400-500
- ขนาดตัวอักษรไม่ควรใหญ่เกินจนดูเหมือนโปสเตอร์

## 20. บทสรุป

Pet Clinic Management System เป็นระบบที่ออกแบบมาเพื่อรองรับงานหลักของคลินิกสัตว์เลี้ยงครบเกือบทุกส่วน ตั้งแต่ข้อมูลลูกค้า สัตว์เลี้ยง นัดหมาย การรักษา วัคซีน ผ่าตัด ใบเสร็จ รายจ่าย ไปจนถึง dashboard และรายงาน

ในเชิงการออกแบบเว็บไซต์ ควรเน้นความเป็น `เว็บแอปสำหรับการทำงานจริง` มากกว่าหน้าโปรโมต โดยใช้แนวทางที่สะอาด น่าเชื่อถือ อ่านง่าย และช่วยให้ผู้ใช้แต่ละกลุ่มทำงานของตนเองได้เร็วที่สุด

เอกสารนี้สามารถใช้เป็นฐานสำหรับ

- สรุปโครงสร้างโปรแกรม
- ใช้อธิบายโปรเจกต์
- ใช้เก็บ requirement/design
- ใช้ต่อยอดการปรับหน้าตาและเก็บงานรอบสุดท้าย

## 21. UI Refresh Plan (2026-07-12)

ส่วนนี้เป็นแผนออกแบบหน้าตาของระบบฉบับใช้งานจริง โดยยึดจากสภาพโปรแกรมปัจจุบัน และข้อสรุปที่ต้องการให้หน้าตาดูเรียบร้อย เป็นงานระบบจริง และไม่มีกลิ่นของงานที่พึ่ง AI ตกแต่งแบบสำเร็จรูป

### 21.1 Design Goal

เป้าหมายของการปรับหน้าตารอบนี้คือ

- ทำให้ระบบดูเป็น `clinic management application` มากกว่า landing page
- ทำให้ฝั่ง `admin` และ `user` ใช้ภาษาดีไซน์เดียวกัน
- เพิ่มความสวยแบบเรียบ สุภาพ ใช้งานจริงได้
- ทำให้ข้อมูลอ่านง่าย สแกนง่าย และ action สำคัญหาเจอง่าย
- ลดจุดที่ดูโล่งเกินไป ดิบเกินไป หรือดูเหมือน template

### 21.2 Visual Direction

บุคลิกของงานออกแบบที่ต้องการ

- `Clean`
- `Modern`
- `Trustworthy`
- `Human`
- `Operational`

คำอธิบาย

- `Clean` คือ พื้นที่หายใจดี ระยะห่างสม่ำเสมอ
- `Modern` คือ ใช้ card, border, shadow และ typography แบบร่วมสมัย
- `Trustworthy` คือ สี สุ้มเสียง และโครงหน้าดูเป็นระบบคลินิกจริง
- `Human` คือ ไม่แข็งจนเกินไป ยังดูเข้าถึงง่ายสำหรับเจ้าของสัตว์เลี้ยง
- `Operational` คือ ทุกหน้าต้องเน้น workflow การใช้งาน ไม่ใช่ decoration

### 21.3 Icon Policy

ข้อกำหนดสำคัญของโปรเจกต์นี้คือ `ใช้ icon จาก code เท่านั้น`

- ไม่ใช้ icon แบบ AI-generated
- ไม่ใช้รูป icon ที่เป็นไฟล์ภาพตกแต่งลอย ๆ
- ใช้ icon library มาตรฐานชุดเดียวทั้งระบบ
- ขนาด icon ต้องสม่ำเสมอใน navigation, button, card และ status
- icon ใช้เพื่อช่วยนำสายตาและลดเวลาในการสแกนข้อมูล

icon ที่แนะนำให้ใช้ในแต่ละเมนู

- Dashboard = `LayoutDashboard` หรือ `BarChart3`
- Reports = `FileText`
- Users = `Users`
- Owners = `UserRound`
- Pets = `PawPrint`
- Veterinarians = `Stethoscope`
- Clinic = `Building2`
- Appointments = `CalendarDays`
- Services = `ClipboardList`
- Treatments = `HeartPulse`
- Surgeries = `Scissors`
- Vaccines = `Syringe`
- Receipts = `Receipt`
- Expenses = `Wallet`
- Logout = `LogOut`

หมายเหตุ: ถ้าในโปรเจกต์มี library อื่นอยู่แล้ว ให้เลือกใช้จาก library เดิมก่อน แต่ต้องคุมให้อยู่ในโทนเดียวกัน

### 21.4 Final Color Direction

ให้ปรับจาก palette เดิมที่ออกม่วงไปทาง `teal + slate + white` มากขึ้น เพื่อให้ดูเป็นคลินิกและลดความรู้สึกเหมือน template

สีหลักที่แนะนำ

- Primary: `#0F766E`
- Primary Hover: `#0B5E57`
- Primary Soft: `#CCFBF1`
- Accent: `#2563EB`
- Background: `#F4F7FB`
- Surface: `#FFFFFF`
- Surface Alt: `#F8FAFC`
- Border: `#D9E2EC`
- Text Main: `#0F172A`
- Text Sub: `#64748B`

สีสถานะ

- Success: `#16A34A`
- Warning: `#D97706`
- Danger: `#DC2626`
- Info: `#2563EB`

แนวทางการใช้สี

- ใช้สีหลักกับปุ่มหลัก เมนู active และจุดที่ต้องการเน้นเพียงบางจุด
- อย่าใช้ gradient ม่วงเต็มจอเป็นพื้นหลังหลักของระบบ
- หน้า data-heavy ควรใช้พื้นขาว เทาอ่อน และเส้นแบ่งที่นุ่มตา
- สีสดใช้กับ status และ call to action เท่านั้น

### 21.5 Typography and Spacing

ข้อกำหนดการจัดวางตัวอักษร

- ฟอนต์หลักแนะนำ `Sarabun` หรือ `Noto Sans Thai`
- หัวข้อหน้าใช้ขนาดใหญ่พอประมาณ ไม่ใช่ hero scale
- subtitle ใช้สีรองและความยาวสั้น
- body text ต้องอ่านง่ายในตารางและฟอร์ม
- line-height ต้องโปร่งพอสำหรับภาษาไทย

ระบบระยะห่าง

- page padding หลัก: `24px - 32px`
- gap ระหว่าง section: `24px`
- gap ระหว่าง card: `16px`
- padding ใน card: `20px - 24px`
- form field spacing: `12px - 16px`

### 21.6 Admin Experience Redesign

ฝั่ง admin ต้องให้ความรู้สึกว่าเป็น `operations console` ของคลินิก

สิ่งที่จะปรับ

- sidebar เข้ม สุภาพ อ่านง่าย และไม่อัดแน่นเกินไป
- จัดเมนูเป็นหมวดชัดเจน เช่น overview, master data, operations, finance
- ทุกหน้ามี page header ที่ประกอบด้วย title + subtitle + action area
- ตารางทุกหน้าต้องมี search, filter และ action button ที่ตำแหน่งคงที่
- ปุ่มสำคัญ เช่น เพิ่มข้อมูล, บันทึก, รับชำระ ต้องเด่นชัด
- ลดการใช้สีหลายโทนในหน้าเดียวโดยไม่จำเป็น

หน้าที่ควรเน้นเป็นพิเศษ

1. Dashboard  
   - จัด card summary ให้เท่ากันทั้งขนาดและ alignment  
   - เพิ่ม visual hierarchy ของกราฟและตัวกรองเดือน/ปี  
   - ทำ recent activity ให้เป็น block แยกชัดเจน

2. Appointments  
   - ฟอร์มเพิ่มนัดหมายต้องอ่านง่ายและไหลลื่น  
   - badge สถานะต้องเห็นชัดทันที  
   - ตารางควรแยกวัน เวลา เจ้าของ และสัตว์เลี้ยงชัดเจน

3. Treatments / Surgeries / Vaccines  
   - ทำฟอร์มให้เป็นกลุ่มข้อมูล เช่น patient, veterinarian, diagnosis, billing  
   - action บันทึกอยู่ตำแหน่งเดิมทุก modal/page  
   - รายการ detail ต้องอ่านง่าย ไม่แน่นเกินไป

4. Receipts / Expenses / Reports  
   - เน้นความชัดของตัวเลข  
   - ใช้ alignment ของจำนวนเงินแบบคอลัมน์ขวา  
   - สถานะการชำระและวิธีชำระต้องแยกด้วย badge หรือ chip

### 21.7 User Experience Redesign

ฝั่ง user ไม่ควรเป็น dashboard หนักแบบ admin แต่ควรเป็น `owner portal`

แนวทาง

- ใช้โทนสว่าง สบายตา และเป็นมิตร
- เมนูมีจำนวนน้อย แต่ชัดเจน
- เน้นข้อมูลของเจ้าของสัตว์และสัตว์เลี้ยงเป็นหลัก
- card ควรมีความนุ่มนวลกว่า admin เล็กน้อย
- รูปแบบข้อมูลต้องเข้าใจง่ายแม้ผู้ใช้ไม่ใช่คนเทคนิค

หน้าที่ควรเน้นเป็นพิเศษ

1. Profile  
   - แสดงข้อมูลเจ้าของแบบ card ชัดเจน  
   - ปุ่มแก้ไขต้องอยู่ในตำแหน่งคาดเดาได้

2. Pets  
   - การ์ดสัตว์เลี้ยงควรมีลำดับชั้นชื่อ, ประเภท, เพศ, อายุ, ข้อมูลแพ้ยา  
   - ปุ่มดูประวัติและแก้ไขต้องเด่นแต่ไม่รก

3. Appointments  
   - แยกนัดหมาย upcoming / past / cancelled ให้ชัด  
   - วันเวลาและสถานะต้องสแกนได้เร็ว  
   - ถ้ามีอีเมลแจ้งเตือน ควรมีข้อความกำกับสั้น ๆ

4. Receipts / History  
   - แสดงลำดับข้อมูลแบบ timeline หรือ clean list  
   - เน้นจำนวนเงิน, วันที่, รายการรักษา และสถานะการชำระ

### 21.9 Things to Avoid

สิ่งที่ไม่ควรใช้ในการตกแต่งรอบนี้

- gradient ม่วงหรือชมพูเต็มหน้า
- card ลอยซ้อนกันหลายชั้น
- icon หลายสไตล์ปนกัน
- ปุ่มสีสดทุกปุ่มในหน้าเดียว
- ตัวอักษรใหญ่เกินจริงแบบ landing page
- หน้าโล่งมากแต่ไม่มีโครงสร้าง
- ภาพตกแต่งที่ไม่เกี่ยวกับคลินิกหรือสัตว์เลี้ยง
- ส่วนประกอบที่ดูเหมือน generated template มากเกินไป

3. ปรับ `AdminLayout.vue` และ `UserLayout.vue`
4. ปรับ component กลาง เช่น table, modal, badge, header
5. ปรับหน้า admin ที่ใช้บ่อยก่อน

6. ปรับหน้า user
