# DFD Level 1 เวอร์ชันที่ควรเป็น

ไฟล์นี้สรุปผัง DFD Level 1 ของระบบ Pet Clinic Management System ให้ตรงกับระบบที่พัฒนาจริงในปัจจุบัน เพื่อใช้เป็นต้นแบบเวลาแก้ใน `draw.io`

---

## 1. External Entity

ใช้ 3 หน่วยงานภายนอกหลัก

1. `ผู้ใช้งานทั่วไป`
2. `เจ้าของสัตว์เลี้ยง`
3. `ผู้ดูแลระบบ / เจ้าหน้าที่คลินิก`

หมายเหตุ:

- `ผู้ใช้งานทั่วไป` ใช้สำหรับดูข้อมูลคลินิกและบริการก่อนเข้าสู่ระบบ
- `เจ้าของสัตว์เลี้ยง` ใช้สำหรับดูข้อมูลของตัวเองหลังเข้าสู่ระบบ
- `ผู้ดูแลระบบ / เจ้าหน้าที่คลินิก` เป็นผู้บันทึกและจัดการข้อมูลธุรกรรมหลักของคลินิก

---

## 3. Process หลัก

### 1.0 แสดงข้อมูลคลินิกและบริการ

หน้าที่:

- แสดงข้อมูลคลินิก
- แสดงรายการบริการและราคา

เชื่อมกับ:

- `ผู้ใช้งานทั่วไป`
- `เจ้าของสัตว์เลี้ยง`
- D1 `ข้อมูลคลินิก`
- D13 `ข้อมูลบริการและราคา`

Data Flow:

- ผู้ใช้งานทั่วไป/เจ้าของสัตว์เลี้ยง -> คำขอดูข้อมูลคลินิกและบริการ -> 1.0
- 1.0 -> ข้อมูลคลินิก -> D1
- 1.0 -> ข้อมูลบริการ -> D13
- 1.0 -> รายละเอียดคลินิกและบริการ -> ผู้ใช้งานทั่วไป/เจ้าของสัตว์เลี้ยง

### 2.0 สมัครสมาชิก

หน้าที่:

- รับข้อมูลสมัครสมาชิก
- บันทึกบัญชีผู้ใช้

เชื่อมกับ:

- `ผู้ใช้งานทั่วไป`
- D2 `ข้อมูลผู้ใช้ระบบ`

Data Flow:

- ผู้ใช้งานทั่วไป -> ข้อมูลสมัครสมาชิก -> 2.0
- 2.0 -> บันทึกผู้ใช้ใหม่ -> D2
- 2.0 -> ผลการสมัครสมาชิก -> ผู้ใช้งานทั่วไป

### 3.0 เข้าสู่ระบบ

หน้าที่:

- ตรวจสอบชื่อผู้ใช้และรหัสผ่าน
- คืนสิทธิ์การใช้งาน

เชื่อมกับ:

- `เจ้าของสัตว์เลี้ยง`
- `ผู้ดูแลระบบ / เจ้าหน้าที่คลินิก`
- D2 `ข้อมูลผู้ใช้ระบบ`

Data Flow:

- ผู้ใช้ -> ข้อมูลเข้าสู่ระบบ -> 3.0
- 3.0 -> ตรวจสอบข้อมูลผู้ใช้ -> D2
- 3.0 -> ผลการเข้าสู่ระบบ/สิทธิ์ใช้งาน -> ผู้ใช้

### 4.0 จัดการข้อมูลเจ้าของสัตว์เลี้ยง

หน้าที่:

- เพิ่ม แก้ไข ค้นหา ข้อมูลเจ้าของสัตว์เลี้ยง

เชื่อมกับ:

- `เจ้าของสัตว์เลี้ยง`
- `ผู้ดูแลระบบ / เจ้าหน้าที่คลินิก`
- D3 `ข้อมูลเจ้าของสัตว์เลี้ยง`
- D2 `ข้อมูลผู้ใช้ระบบ`

Data Flow:

- เจ้าของสัตว์เลี้ยง/เจ้าหน้าที่ -> ข้อมูลเจ้าของสัตว์เลี้ยง -> 4.0
- 4.0 -> บันทึก/แก้ไขข้อมูลเจ้าของ -> D3
- 4.0 -> อ่านข้อมูลผู้ใช้ประกอบ -> D2
- 4.0 -> ผลการจัดการข้อมูลเจ้าของ -> ผู้เกี่ยวข้อง

หมายเหตุ:

- ไม่ใช้ `owner_address` แล้ว ถ้าจะเขียน data dictionary ให้ลบ field นี้ออก

### 5.0 จัดการข้อมูลสัตว์เลี้ยง

หน้าที่:

- เพิ่ม แก้ไข ค้นหา ข้อมูลสัตว์เลี้ยง

เชื่อมกับ:

- `เจ้าของสัตว์เลี้ยง`
- `ผู้ดูแลระบบ / เจ้าหน้าที่คลินิก`
- D4 `ข้อมูลสัตว์เลี้ยง`
- D3 `ข้อมูลเจ้าของสัตว์เลี้ยง`

Data Flow:

- เจ้าของสัตว์เลี้ยง/เจ้าหน้าที่ -> ข้อมูลสัตว์เลี้ยง -> 5.0
- 5.0 -> ตรวจสอบเจ้าของสัตว์ -> D3
- 5.0 -> บันทึก/แก้ไขข้อมูลสัตว์เลี้ยง -> D4
- 5.0 -> ผลการจัดการข้อมูลสัตว์เลี้ยง -> ผู้เกี่ยวข้อง

### 6.0 จัดการข้อมูลการนัดหมาย

หน้าที่:

- สร้าง แก้ไข ยืนยัน ยกเลิก การนัดหมาย
- เจ้าของสัตว์เลี้ยงเข้ามาดูสถานะนัดหมายของตัวเองได้

เชื่อมกับ:

- `เจ้าของสัตว์เลี้ยง`
- `ผู้ดูแลระบบ / เจ้าหน้าที่คลินิก`
- D5 `ข้อมูลการนัดหมาย`
- D4 `ข้อมูลสัตว์เลี้ยง`
- D3 `ข้อมูลเจ้าของสัตว์เลี้ยง`

Data Flow:

- เจ้าหน้าที่ -> ข้อมูลนัดหมาย -> 6.0
- 6.0 -> ตรวจสอบสัตว์เลี้ยง -> D4
- 6.0 -> อ่านข้อมูลเจ้าของสัตว์เลี้ยง -> D3
- 6.0 -> บันทึก/แก้ไขนัดหมาย -> D5
- เจ้าของสัตว์เลี้ยง -> คำขอดูข้อมูลนัดหมาย -> 6.0
- 6.0 -> รายการนัดหมาย/สถานะนัดหมาย -> เจ้าของสัตว์เลี้ยง
- 6.0 -> ผลการจัดการนัดหมาย -> เจ้าหน้าที่

หมายเหตุสำคัญ:

- เวอร์ชันปัจจุบัน **ไม่มีการจองนัดจากฝั่ง user**
- คนที่สร้างนัดหมายคือ `ผู้ดูแลระบบ / เจ้าหน้าที่คลินิก`
- ถ้าต้องการใส่ใน DFD ให้เขียนว่า “ดูข้อมูลนัดหมาย” สำหรับเจ้าของสัตว์เลี้ยง ไม่ใช่ “จองนัดหมาย”

### 7.0 บันทึกการรักษาและหัตถการ

หน้าที่:

- บันทึกอาการ วินิจฉัย วันที่รักษา
- บันทึกสัตวแพทย์ผู้รักษา
- บันทึกรายการบริการที่ใช้ในการรักษาแต่ละครั้ง

เชื่อมกับ:

- `ผู้ดูแลระบบ / เจ้าหน้าที่คลินิก`
- `เจ้าของสัตว์เลี้ยง`
- D6 `ข้อมูลการรักษา`
- D15 `รายละเอียดการรักษา`
- D4 `ข้อมูลสัตว์เลี้ยง`
- D13 `ข้อมูลบริการและราคา`
- D14 `ข้อมูลสัตวแพทย์`

Data Flow:

- เจ้าหน้าที่ -> ข้อมูลการรักษา -> 7.0
- 7.0 -> ตรวจสอบสัตว์เลี้ยง -> D4
- 7.0 -> อ่านข้อมูลสัตวแพทย์ -> D14
- 7.0 -> อ่านข้อมูลบริการ -> D13
- 7.0 -> บันทึกข้อมูลการรักษา -> D6
- 7.0 -> บันทึกรายละเอียดการรักษา -> D15
- เจ้าของสัตว์เลี้ยง -> คำขอดูประวัติการรักษา -> 7.0
- 7.0 -> ประวัติการรักษา -> เจ้าของสัตว์เลี้ยง

หมายเหตุ:

- ใน DFD ใช้คำว่า `ข้อมูลสัตวแพทย์`
- ในฐานข้อมูลจริงเก็บ `vet_id` ในตารางธุรกรรม และค่อย join ไปแสดงชื่อหมอ

### 8.0 บันทึกประวัติวัคซีน

- บันทึกข้อมูลวัคซีนที่สัตว์เลี้ยงได้รับ
- ระบุสัตวแพทย์ผู้รับผิดชอบ

เชื่อมกับ:

- `ผู้ดูแลระบบ / เจ้าหน้าที่คลินิก`
- `เจ้าของสัตว์เลี้ยง`
- D8 `ข้อมูลวัคซีน`
- D4 `ข้อมูลสัตว์เลี้ยง`
- D13 `ข้อมูลบริการและราคา`
- D14 `ข้อมูลสัตวแพทย์`

Data Flow:

- เจ้าหน้าที่ -> ข้อมูลวัคซีน -> 8.0
- 8.0 -> ตรวจสอบสัตว์เลี้ยง -> D4
- 8.0 -> อ่านข้อมูลบริการ -> D13
- 8.0 -> อ่านข้อมูลสัตวแพทย์ -> D14
- 8.0 -> บันทึกประวัติวัคซีน -> D8
- เจ้าของสัตว์เลี้ยง -> คำขอดูประวัติวัคซีน -> 8.0
- 8.0 -> ประวัติวัคซีน -> เจ้าของสัตว์เลี้ยง

### 9.0 บันทึกการผ่าตัด

### 10.0 บันทึกการชำระเงินและออกใบเสร็จ

เชื่อมกับ:

Data Flow:

- 14.0 -> บันทึกข้อมูลสัตวแพทย์ -> D14
- 14.0 -> ผลการจัดการข้อมูลหลัก -> เจ้าหน้าที่

## 4. จุดที่ต้องระวังเวลาแก้ใน draw.io

### เรื่องการนัดหมาย

- ห้ามวาดเป็น `เจ้าของสัตว์เลี้ยง -> จองนัดหมาย -> ระบบ`
- ให้แก้เป็น `เจ้าหน้าที่คลินิก -> จัดการข้อมูลนัดหมาย -> ระบบ`
- เจ้าของสัตว์เลี้ยงมีได้แค่
  - ดูข้อมูลนัดหมาย
  - ดูสถานะนัดหมาย
  - รับแจ้งเตือนการนัดหมาย

### เรื่องการรักษา

- ต้องมี `ข้อมูลสัตวแพทย์` เชื่อมกับ process การรักษา
- ต้องมี `รายละเอียดการรักษา` เป็น data store แยก
- ถ้าจะเขียน subprocess ของ 7.0 สามารถแตกได้เป็น:
  - 7.1 บันทึกข้อมูลการรักษา
  - 7.2 บันทึกรายการบริการที่ใช้
  - 7.3 คำนวณค่าใช้จ่ายรวม

### เรื่องใบเสร็จ

- ต้องมีทั้ง `ข้อมูลใบเสร็จรับเงิน` และ `รายละเอียดใบเสร็จ`
- ไม่ควรใช้ data store ใบเสร็จตัวเดียวถ้าในระบบจริงแยก header/detail

### เรื่องชื่อหมอ

- ใน DFD ใช้คำว่า `ข้อมูลสัตวแพทย์`
- ในฐานข้อมูลจริงอ้างอิงด้วย `vet_id`
- ใช้แนวคิดว่า process อ่านข้อมูลสัตวแพทย์จาก D14 เพื่อบันทึกหรือแสดงผล

## 5. ถ้าอยากวาดแบบดูง่ายในหน้าเดียว

ลำดับการวาง process แนะนำ:

1. วาง `1.0 2.0 3.0` ไว้แถวบน
2. วาง `4.0 5.0 6.0` ไว้แถวกลางซ้าย
3. วาง `7.0 8.0 9.0 10.0` ไว้แถวกลาง/ล่าง
4. วาง `11.0 12.0 13.0 14.0` ไว้แถวล่าง
5. วาง Data Store ทั้ง 15 ตัวไว้ด้านขวาหรือด้านล่างแล้วลากเส้นเข้า process ที่เกี่ยวข้อง

## 6. สรุปสั้นที่สุดสำหรับเอาไปแก้เอกสาร

- ฝั่ง user = ดูข้อมูลเป็นหลัก
- ฝั่ง admin/เจ้าหน้าที่ = เป็นคนบันทึกธุรกรรมหลัก
- การนัดหมาย = คลินิกสร้าง ไม่ใช่ user จองเอง
- การรักษา = ต้องมีสัตวแพทย์และรายละเอียดการรักษา
- การเงิน = ต้องมีใบเสร็จและรายละเอียดใบเสร็จ
- DFD ควรอิงฐานข้อมูลจริง 15 ตาราง
